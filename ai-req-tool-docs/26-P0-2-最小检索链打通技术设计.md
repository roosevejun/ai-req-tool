# P0-2 最小检索链打通技术设计

## 1. 目标

本设计文档用于把“项目知识库”从当前的：

- 可沉淀
- 可处理
- 可查看

推进到：

- 可检索
- 可被 AI 复用

本阶段不追求完整知识平台，而是优先打通一条最小可用链路：

**知识文档 -> chunk -> embedding -> 向量检索 -> 召回片段 -> AI 上下文注入**

---

## 2. 当前现状

当前工程已经具备：

- `km_document`
- `km_document_asset`
- `km_document_chunk`
- `km_document_task`

并且在：

- [KnowledgeDocumentProcessingService.java](c:\javaDevelop\ai-req-tool\docgen-webapp\backend\src\main\java\com\tongtu\docgen\project\service\KnowledgeDocumentProcessingService.java)
- [KnowledgeDocumentService.java](c:\javaDevelop\ai-req-tool\docgen-webapp\backend\src\main\java\com\tongtu\docgen\project\service\KnowledgeDocumentService.java)

里已经完成：

- 文本清洗

- chunk 落表

当前缺失的是：

- chunk embedding 生成
- 向量数据存储
- 检索服务
- 下游 AI 引用链

---

## 3. 本阶段范围

## 3.1 要做的

- 为知识 chunk 增加向量存储能力
- 使用 `pgvector` 实现最小向量检索
- 增加知识检索服务
- 提供按项目范围召回知识片段能力
- 在需求 AI 工作台中接入最小召回

## 3.2 暂时不做的

- 多路检索重排
- 混合检索排序优化
- 独立向量数据库
- 复杂 query rewrite
- 多租户知识库治理
- 权重学习

---

## 4. 方案选择

## 4.1 为什么优先选择 pgvector

本阶段推荐优先使用 PostgreSQL + pgvector，而不是独立向量库。

原因：

1. 当前系统已经基于 PostgreSQL
2. 部署成本最低
3. 便于和 `km_document_chunk` 直接关联
4. 更适合先验证“知识库检索是否真的提升 AI 质量”

结论：

**P0-2 先用 pgvector 做最小闭环，后续再视规模决定是否迁移到 Qdrant / Milvus。**

## 5. 数据模型设计

## 5.1 扩展 km_document_chunk

当前 `km_document_chunk` 已有：

- `embedding_status`
- `vector_ref`

本阶段建议新增字段：

- `embedding_model VARCHAR(64) NULL`
- `embedding_vector vector(1536) NULL`
- `embedded_at TIMESTAMP NULL`

说明：

- `1536` 是建议的首版维度，可根据最终 embedding 模型调整
- 如果后续模型维度变化，需要做版本兼容或新列策略

## 5.2 新增索引

建议增加：

- `idx_km_doc_chunk_doc_status`
- 向量索引 `ivfflat`

示意：

```sql
CREATE EXTENSION IF NOT EXISTS vector;

ALTER TABLE km_document_chunk
    ADD COLUMN IF NOT EXISTS embedding_model VARCHAR(64) NULL;

ALTER TABLE km_document_chunk
    ADD COLUMN IF NOT EXISTS embedding_vector vector(1536) NULL;

ALTER TABLE km_document_chunk
    ADD COLUMN IF NOT EXISTS embedded_at TIMESTAMP NULL;

CREATE INDEX IF NOT EXISTS idx_km_doc_chunk_doc_status
    ON km_document_chunk (document_id, embedding_status);

CREATE INDEX IF NOT EXISTS idx_km_doc_chunk_vector_ivfflat
    ON km_document_chunk
    USING ivfflat (embedding_vector vector_cosine_ops)
    WITH (lists = 100);
```

建议新增：

4. 组装召回结果

建议提供的方法：

- `embedPendingChunks(int limit)`
- `embedChunk(Long chunkId)`
- `searchProjectKnowledge(Long projectId, String query, int topK)`

## 6.3 新增 Mapper 能力

建议扩展：

- [KnowledgeDocumentChunkMapper.java](c:\javaDevelop\ai-req-tool\docgen-webapp\backend\src\main\java\com\tongtu\docgen\project\mapper\KnowledgeDocumentChunkMapper.java)

检索 SQL 方向：

- 通过 `km_document_chunk -> km_document` 关联项目/需求范围
- 过滤：
  - `retrievable = true`
  - `embedding_status = 'READY'`
- 按向量距离排序

## 7. 检索数据流

## 7.1 文档入库后流程

1. 创建 `km_document`
2. 文本清洗
3. chunk 切片
4. chunk 初始状态：
   - `embedding_status = PENDING`
5. 后台 worker 调用 embedding
6. 写入 `embedding_vector`
7. 状态更新为：
   - `embedding_status = READY`

## 7.2 AI 查询时流程

1. 构造查询文本
2. 为查询文本生成 embedding
3. 在项目范围内检索 topK chunk
4. 过滤低质量结果
5. 把命中 chunk 组装成知识上下文文本
6. 注入下游 AI prompt

响应示例：

```json

    {
      "documentId": 12,
      "chunkId": 88,
      "score": 0.91,
      "chunkText": "..."
    }
  ]
}
```

这个接口主要用于：

- 联调验证
- 人工检查召回质量

## 9. 与需求 AI 工作台的接入方式

## 9.1 首版接入点

优先接在：

- [RequirementDocgenService.java](c:\javaDevelop\ai-req-tool\docgen-webapp\backend\src\main\java\com\tongtu\docgen\project\service\RequirementDocgenService.java)

接入位置：

- 创建需求 AI 会话时
- 继续对话时
- 生成 PRD 时

## 9.2 查询文本构造

建议首版用简单策略：

- `需求标题 + 需求摘要`
- 如果是继续对话，再加上最近一轮 user message

示意：

```text
需求标题
需求摘要
最近一轮用户输入
```

## 9.3 Prompt 注入方式

建议增加独立上下文块，而不是直接拼进用户原文。

格式示意：

```text
[Project Knowledge Context]
1. ...
2. ...
3. ...
```

这样更利于：

- 控制长度
- 调试
- 后续替换策略

## 10. 首版召回策略

## 10.1 检索范围

首版建议：

- 只按 `project_id` 检索
- 如果后续有 requirement 级知识，再叠加 requirement 范围

## 10.2 TopK

建议默认：

- `topK = 5`

## 10.3 长度控制

建议：

- 单 chunk 限制 500-800 字符
- 总上下文限制 2500-3000 字符

## 10.4 结果过滤

建议最小过滤：

- 仅 `retrievable = true`
- 仅 `embedding_status = READY`
- 去重相同 document 的重复 chunk

## 12. 验收标准

满足以下条件可认为 P0-2 完成：

1. 新增知识文档被切片后，可进一步生成 embedding
2. 项目范围内可执行一次有效 chunk 检索
3. 检索结果可注入需求 AI 工作台
4. 需求 AI 对话/生成已能利用知识库上下文
5. 检索链路失败时，不影响主业务流程

## 13. 风险与注意点

## 13.1 pgvector 依赖

风险：

- 部署环境未安装 pgvector

应对：

- 升级脚本中显式 `CREATE EXTENSION IF NOT EXISTS vector`
- 部署文档增加环境要求

## 13.2 向量维度绑定模型

风险：

- embedding 模型一旦变化，维度可能不兼容

应对：

- 增加 `embedding_model`
- 首版固定单模型

## 13.3 召回噪音

风险：

- chunk 命中不准，反而污染 AI 上下文

应对：

- 首版先控制 topK 和长度
- 后续再做 rerank

## 13.4 请求成本和时延

风险：

- embedding 增加额外 LLM 调用成本

应对：

- 仅对新 chunk 做 embedding
- embedding 任务异步化

---

## 14. 结论

P0-2 的本质不是“再加一个表字段”，而是把知识库从“静态资产”变成“可参与 AI 推理的上下文系统”。

本阶段最合理的路线是：

**先用 pgvector 打通最小检索链，再把结果接到需求 AI 工作台，优先验证真实业务价值。**

这一步完成之后，项目知识库才真正开始具备“可复用”的意义。
