# 当前工程约束与需求基线（docgen-webapp）

## 1. 项目定位
- `docgen-webapp` 是 AI 需求整理工具，不是 ThingsBoard 业务系统本身。
- ThingsBoard 仅作为示例需求之一，系统目标是支持通用需求收集、澄清和 PRD 生成。

## 2. 核心业务目标
- 通过对话方式收集用户需求。
- AI 对需求进行分析，识别信息缺口并继续追问。
- 在“信息充分”后生成 `01-PRD-Agent需求文档.md`。
- 支持基于上一版 PRD 的增量修订，形成新版本 PRD（循环迭代）。

## 3. 会话与状态模型
- 以 `jobId` 作为一次需求会话标识。
- 关键状态：`CLARIFYING` / `READY` / `COMPLETED`。
- 后端保存：
- 对话历史 `chatHistory`
- 已确认项 `confirmedItems`
- 未确认项 `unconfirmedItems`
- 当前待确认问题 `pendingQuestion`
- 规则：
- `unconfirmedItems` 非空时，不允许生成 PRD。
- `unconfirmedItems` 清零后，才允许生成 PRD。

## 4. 前端交互要求
- “未确认项”必须由后端返回，前端不得硬编码默认项。
- 采用“逐条问答卡片”模式降低输入压力：
- 单题展示（当前未确认项）
- `上一题/下一题`
- `发送当前题答案`
- 保留 `合并全部并发送` 作为补充模式。
- 前端状态展示与后端生成校验必须一致，避免“页面显示可生成但后端拦截”。

## 5. PRD 生成与元数据规范
- 生成后文档顶部自动写入服务端生成时间（Asia/Shanghai）。
- `文档版本` 自动递增：
- 首次默认 `V1.0`
- 后续 `V1.0 -> V1.0.1 -> V1.0.2 ...`
- `日期` 自动写入当前生成日期，格式：`yyyy/M/d`（如 `2026/3/24`）。

## 6. 导出与编码兼容性要求
- 支持下载 Markdown。
- `Content-Disposition` 需同时包含：
- `filename`（ASCII 安全名）
- `filename*`（UTF-8 编码名）
- 避免 Tomcat 对中文响应头报错（`outside the permitted range of 0 to 255`）。

## 7. LLM 接入与配置约束
- 支持 OpenAI 兼容网关（`/chat/completions`）。
- API Key 读取优先级：
- `AGENT_ROUTER_TOKEN`
- `OPENAI_API_KEY`
- `agent.llm.openai.apiKey`（本地配置）
- 兼容网关返回差异（`message.content` 可能为字符串/数组/对象）。
- 无 key 或 mock 模式下可降级到 mock 流程，保证前后端联调可进行。

## 8. 错误处理与可观测性
- 统一返回结构：`code/message/data/traceId`。
- 前端展示可读错误信息；后端记录异常堆栈并携带 `traceId`。
- 对关键失败（401/429/网络超时/握手失败）需可快速定位配置、网络或额度问题。

## 9. 当前阶段实现优先级
1. 对话澄清闭环稳定（未确认项引导、状态一致性）。
2. PRD 生成质量与版本化稳定（版本、日期、生成时间自动化）。
3. 导出稳定性（中文文件名兼容）。
4. 模型网关兼容与配置健壮性（减少环境问题带来的阻塞）。

## 10. 非目标（当前迭代不做）
- 不实现具体业务系统（如 ThingsBoard 车辆大屏本体）。
- 不替代团队正式项目管理系统（仅作为需求整理与文档生产前置工具）。
- 暂不引入复杂持久化与多租户权限体系（可后续演进）。
