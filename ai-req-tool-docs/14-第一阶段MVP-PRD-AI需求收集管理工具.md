# 第一阶段 MVP PRD：AI需求收集管理工具

## 1. 背景

在项目早期，需求信息通常分散在口头沟通、聊天记录、会议纪要、零散文档和个人理解中，缺少统一入口与结构化沉淀方式。结果往往是：

- 需求表达不完整
- 多轮沟通反复确认
- 文档整理耗时长
- 需求版本难追踪
- 后续研发接收的信息不稳定

因此，第一阶段 MVP 的目标不是覆盖整个项目管理流程，而是先验证一条最核心闭环：

`项目 -> 需求 -> AI澄清 -> 文档生成 -> 版本沉淀`

---

## 2. MVP目标

第一阶段 MVP 要验证的核心问题是：

系统是否能够帮助团队把一个项目中的初始需求，更快地收集、澄清、整理，并形成可查看、可导出、可追踪的正式需求文档。

### 2.1 核心目标

1. 提供统一的项目与需求收集入口
2. 支持 AI 参与需求澄清
3. 支持生成结构化需求文档
4. 支持需求版本沉淀与导出
5. 为后续多人协同和扩展保留基础结构

### 2.2 非核心目标

本阶段不追求：

- 完整项目流程管理
- 复杂审批流
- 自动任务拆分
- 接口设计自动生成
- 多渠道自动采集

---

## 3. 目标用户

### 3.1 主要用户

- 产品经理
- 需求分析师
- 项目经理
- 实施经理

### 3.2 次要用户

- 业务方
- 研发负责人
- 测试负责人

### 3.3 使用场景

1. 项目刚启动，团队需要把业务想法整理成正式需求
2. 已有需求草稿，但内容不完整，需要通过 AI 继续澄清
3. 需求多次修订，需要保留历史版本并导出当前版本

---

## 4. 用户故事

### 4.1 项目与需求创建

作为产品经理，我希望先创建项目，再在项目下创建具体需求，以便按项目统一管理需求内容。

### 4.2 AI辅助澄清

作为需求分析人员，我希望系统能根据已有描述主动追问缺失信息，帮助我更快把需求说清楚。

### 4.3 文档生成

作为项目经理，我希望在需求澄清完成后快速生成标准化需求文档，减少手工整理时间。

### 4.4 版本沉淀

作为团队成员，我希望每次需求生成后的文档能保存为版本，便于回溯和导出。

---

## 5. MVP范围

### 5.1 范围内

1. 登录与基础权限控制
2. 项目创建、列表、详情、编辑
3. 需求创建、列表、详情、编辑
4. 基于需求发起 AI 工作台会话
5. AI 多轮澄清
6. 生成需求文档 PRD
7. 保存需求版本
8. 查看需求版本详情
9. 导出需求版本 Markdown
10. 系统管理中的基础用户、角色、权限维护

### 5.2 范围外

1. 项目成员协同分工完整流程
2. 完整的会话消息持久化与恢复
3. 文档在线协同编辑
4. 审批流与发布流
5. 多渠道输入采集
6. 自动拆分研发任务
7. 测试用例自动生成

---

## 6. 核心业务流程

### 6.1 主流程

1. 用户登录系统
2. 创建项目
3. 在项目下创建需求
4. 进入需求 AI 工作台
5. 输入业务描述，必要时附带上一版 PRD
6. AI 发起澄清问题，用户多轮补充信息
7. 当需求信息足够完整时，生成需求文档
8. 系统保存需求版本
9. 用户查看版本详情或导出 Markdown

### 6.2 简化流程图文字版

`登录 -> 项目 -> 需求 -> AI澄清 -> 生成PRD -> 保存版本 -> 查看/导出`

## 8. 核心功能需求

### 8.1 登录与身份识别

系统应支持用户登录，并在登录后识别当前用户身份，用于权限控制和业务操作留痕。

具体要求：

1. 支持用户名密码登录
2. 登录成功后返回 token
3. 前端后续请求自动带 token
4. 失效 token 自动跳转登录页

### 8.2 项目管理

系统应支持项目的基础管理能力。

具体要求：

1. 可创建项目
2. 可查看项目列表
3. 可查看项目详情
4. 可编辑项目基础信息
5. 项目与需求形成归属关系

### 8.3 需求管理

系统应支持需求的基础管理能力。

具体要求：

1. 可在项目下创建需求
2. 系统自动生成需求编号
3. 可查看需求详情
4. 可编辑需求基础信息
5. 需求与版本形成一对多关系

### 8.4 AI澄清工作台

系统应支持围绕某一需求发起 AI 会话，并持续补齐关键信息。

具体要求：

1. 可基于需求发起一轮 AI 会话
2. 可输入业务描述
3. 可选输入上一版 PRD
4. AI 返回：
   - 已确认项
   - 未确认项
   - 当前待确认问题
   - 聊天记录
5. 用户可通过自然语言或结构化方式补充信息
6. AI 可多轮迭代更新澄清状态

### 8.5 文档生成

系统应在需求满足生成条件后，生成正式需求文档。

具体要求：

1. 当未确认项为空时，可执行生成
2. 生成结果为 Markdown 文档
3. 文档应包含必要的结构化内容
4. 文档应支持版本号与生成时间标记

### 8.6 版本管理

系统应保存每次生成结果为一个需求版本。

具体要求：

1. 生成成功后保存版本
2. 支持查看版本列表
3. 支持查看单个版本详情
4. 支持标识当前版本
5. 支持导出指定版本 Markdown

### 8.7 系统管理

系统应支持基础的用户、角色、权限维护。

具体要求：

1. 支持用户创建与编辑
2. 支持角色创建与编辑
3. 支持权限创建与编辑
4. 支持用户绑定角色
5. 支持角色绑定权限

---

## 9. AI功能要求

### 9.1 AI的输入

AI 生成与澄清应至少基于以下输入：

1. 当前业务描述
2. 当前对话历史
3. 已确认项与未确认项
4. 可选的上一版 PRD
5. 文档模板

### 9.2 AI的输出

AI 在澄清阶段应输出：

1. assistantMessage
2. confirmedItems
3. unconfirmedItems
4. pendingQuestion

AI 在生成阶段应输出：

1. prdMarkdown

### 9.3 AI的价值要求

AI 输出不应只是泛化模板，而应体现当前需求的具体业务内容。

### 9.4 AI的边界要求

AI 可以辅助澄清和生成，但不能替代人工做最终业务确认。

## 12. 权限要求

MVP 阶段权限要求如下：

1. 登录后可访问基础查看能力
2. 创建项目需 `PROJECT:CREATE`
3. 编辑项目需 `PROJECT:EDIT`
4. 创建需求需 `REQUIREMENT:CREATE`
5. 编辑需求需 `REQUIREMENT:EDIT`
6. 参与 AI 澄清需 `REQUIREMENT:AI_CHAT`
7. 生成 PRD 需 `REQUIREMENT:GENERATE_PRD`
8. 查看版本需 `REQUIREMENT:VERSION_VIEW`
9. 系统管理接口需 `ROLE:ADMIN`

---

## 13. 验收标准

### 13.1 业务闭环验收

- 可以成功登录系统
- 可以创建项目
- 可以在项目下创建需求
- 可以进入需求 AI 工作台
- 可以完成至少一轮 AI 澄清
- 可以生成一版需求文档
- 可以看到生成后的版本记录
- 可以导出版本 Markdown

### 13.2 数据验收

- 项目数据成功落库
- 需求数据成功落库
- 需求版本数据成功落库
- AI 会话摘要状态成功落库

### 13.3 权限验收

- 未登录用户无法访问受保护接口
- 无权限用户无法调用受限接口
- 管理员可访问系统管理功能

### 13.4 文档验收

- 生成文档应为 Markdown
- 文档应包含基本结构化内容
- 导出文件应可正常打开

---

## 14. 非功能要求

### 14.1 可用性

- 页面流程应清晰，用户能在较少学习成本下完成核心操作

### 14.2 可追踪性

- 所有关键操作应能追溯到用户、需求、版本、生成时间

### 14.3 可扩展性

- 后续应能扩展完整会话持久化、审批流、任务拆分等能力

### 14.4 稳定性

- 常见错误应返回统一错误信息
- token 失效与权限不足应有明确反馈

## 16. 当前阶段建议

从产品推进角度，第一阶段接下来最值得关注的不是继续堆功能，而是聚焦以下问题：

1. AI 澄清是否真正有效
2. 生成文档是否足够贴近真实业务输入
3. 需求版本是否具有管理价值
4. 项目、需求、版本、权限之间的主流程是否顺畅

因此，第一阶段的重点应是：

把核心闭环做稳、做顺、做出真实价值感。
