## 1. 典型业务调用链

下面按真实开发中最常走的几条链路，串起“页面操作 -> 接口 -> 服务 -> 数据库”。

### 1.1 登录

- 前端页面：[docgen-webapp/frontend/src/views/LoginPage.vue](../docgen-webapp/frontend/src/views/LoginPage.vue)
- 请求接口：`POST /api/system/auth/login`
- 后端入口：[docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/api/SystemAuthController.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/api/SystemAuthController.java)
- 核心服务：[docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/service/SystemAuthService.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/service/SystemAuthService.java)
- 认证工具：
  - [docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/util/PasswordUtil.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/util/PasswordUtil.java)
  - [docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/util/TokenUtil.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/util/TokenUtil.java)
- 数据访问：
  - [docgen-webapp/backend/src/main/resources/mapper/system/SystemUserMapper.xml](../docgen-webapp/backend/src/main/resources/mapper/system/SystemUserMapper.xml)

流程说明：

1. 前端提交用户名和密码。
2. `SystemAuthController` 调用 `SystemAuthService.login(...)`。
3. `SystemAuthService` 查询 `sys_user`，校验账号状态与密码。
4. 登录成功后生成自定义 HMAC token，并加载当前用户角色、权限。
5. 前端把 token 与用户信息保存到 `localStorage`。

涉及数据表：

- `sys_user`
- `sys_user_role`
- `sys_role_permission`
- `sys_role`
- `sys_permission`

### 1.2 访问受保护接口

- 前端入口：[docgen-webapp/frontend/src/main.ts](../docgen-webapp/frontend/src/main.ts)
- 前端本地认证状态：[docgen-webapp/frontend/src/auth.ts](../docgen-webapp/frontend/src/auth.ts)
- 后端拦截器配置：[docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/config/WebConfig.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/config/WebConfig.java)
- 后端拦截器实现：[docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/security/AuthInterceptor.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/system/security/AuthInterceptor.java)

流程说明：

1. axios 请求拦截器自动附加 `Authorization: Bearer <token>`。
2. 后端 `AuthInterceptor` 拦截 `/api/**`。
3. 解析并校验 token。
4. 加载用户角色与权限，写入 `UserContextHolder`。
5. 若接口标注了 `@RequiredPermission`，则执行权限校验。

- 核心服务：[docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/service/RequirementDocgenService.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/service/RequirementDocgenService.java)
- 适配委托：[docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/service/DocGenControllerDelegate.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/service/DocGenControllerDelegate.java)
- AI 核心：
  - [docgen-webapp/backend/src/main/java/com/tongtu/docgen/service/DocGenService.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/service/DocGenService.java)

  - [docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/mapper/RequirementDocgenMapper.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/mapper/RequirementDocgenMapper.java)

流程说明：

1. 前端传入业务描述与可选的上一版 PRD。

7. 将会话摘要信息写入 `rm_requirement_chat_session`。

涉及数据表：

- `rm_requirement`
- `rm_requirement_chat_session`

### 1.6 AI 对话澄清

- 前端组件：[docgen-webapp/frontend/src/components/DocGenPage.vue](../docgen-webapp/frontend/src/components/DocGenPage.vue)
- 请求接口：`POST /api/requirements/{requirementId}/docgen/jobs/{jobId}/chat`
- 后端入口：[docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/api/RequirementDocgenController.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/api/RequirementDocgenController.java)
- 核心服务：[docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/service/RequirementDocgenService.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/project/service/RequirementDocgenService.java)
- AI 核心：[docgen-webapp/backend/src/main/java/com/tongtu/docgen/service/DocGenService.java](../docgen-webapp/backend/src/main/java/com/tongtu/docgen/service/DocGenService.java)

流程说明：

1. 前端发送补充说明或结构化回答。
2. 后端先校验 `requirementId + jobId` 对应会话存在。
3. `DocGenService.chat(...)` 继续调用 LLM。
4. 新的确认项、待确认项、待回答问题回写到 `rm_requirement_chat_session`。

当前实现特点：

- 详细聊天历史主要存在 `DocGenService` 的内存 `jobs` 中。
- 数据库中虽然已设计 `rm_requirement_chat_message` 表，但当前代码还未真正写入聊天消息。

## 3. 当前工程值得特别留意的点

### 3.1 AI 会话状态目前是“内存 + 部分落库”

- `DocGenService` 使用内存 `jobs` 保存详细对话过程。
- `rm_requirement_chat_session` 仅保存会话摘要状态。
- 服务重启后，内存中的 AI 会话会丢失。

### 3.2 聊天消息表已建但未充分使用

- 数据库中已有 `rm_requirement_chat_message`。
- 当前代码尚未把完整聊天历史写入该表。
- 这意味着“可恢复会话”和“聊天审计”还未完全实现。

### 3.3 前端采用轻状态、直连接口模式

- 多数页面直接使用 axios 调接口。
- 当前优点是容易理解、改动成本低。
- 后期若页面增多，可能需要抽离统一 API 层和状态管理层。

## 5. 核心类关系图文字版

下面用“谁依赖谁、谁调用谁”的方式，把关键类关系串起来。

### 5.1 后端整体分层

后端大致可以看成 5 层：

1. 启动与配置层
2. 横切基础层
3. 认证权限层
4. 项目需求业务层
5. AI 文档生成层

### 5.2 启动与配置层

- `DocGenApplication`
  - Spring Boot 启动入口
- `WebConfig`
  - 注册 `AuthInterceptor`
- `OpenApiConfig`
  - 配置 Swagger/OpenAPI 与权限扩展

关系说明：

- `DocGenApplication` 启动整个 Spring 容器。
- `WebConfig` 把认证拦截器接入请求链。
- `OpenApiConfig` 不参与业务处理，但影响接口文档展示。

### 5.3 横切基础层

- `ApiResponse<T>`
  - 所有接口统一返回结构
- `GlobalExceptionHandler`
  - 统一异常收口
- `UserContext`
  - 当前登录用户上下文
- `UserContextHolder`
  - 使用 `ThreadLocal` 保存当前请求用户信息

关系说明：

- Controller 返回时统一包一层 `ApiResponse`。
- 认证成功后，`AuthInterceptor` 将用户信息写入 `UserContextHolder`。
- 业务代码通过 `AccessGuard` 或 `UserContextHolder` 获取当前用户。
- 所有未捕获的业务异常最终由 `GlobalExceptionHandler` 转成标准响应。

### 5.4 认证权限层

核心类：

- `AuthInterceptor`
- `AccessGuard`
- `SystemAuthController`
- `SystemAuthService`
- `TokenUtil`
- `PasswordUtil`
- `SystemUserMapper`
- `SystemRoleMapper`
- `SystemPermissionMapper`

关系图文字版：

- `SystemAuthController`
  - 调用 `SystemAuthService`
- `SystemAuthService`
  - 调用 `SystemUserMapper`
  - 调用 `PasswordUtil`
  - 调用 `TokenUtil`
- `AuthInterceptor`
  - 调用 `TokenUtil` 验 token
  - 调用 `SystemAuthService.loadProfile(...)`
  - 写入 `UserContextHolder`
  - 调用内部权限校验逻辑
- `AccessGuard`
  - 读取 `UserContextHolder`
  - 提供“必须登录”与“必须具备角色”的快速校验

理解重点：

- 登录阶段是 `Controller -> Service -> Mapper/Util`。
- 接口访问阶段是 `Interceptor -> TokenUtil + SystemAuthService`。
- 权限判断有两层：
  - 拦截器根据 `@RequiredPermission` 做细粒度权限判断
  - 业务代码中偶尔再用 `AccessGuard.requireLogin()` 做兜底

### 5.5 项目需求业务层

- `ProjectService`

- `rm_requirement_chat_message`
  - 某次会话下的消息明细

字段关系重点：

- `rm_requirement_chat_session.requirement_id -> rm_requirement.id`
- `rm_requirement_chat_session.created_by -> sys_user.id`
- `rm_requirement_chat_message.session_id -> rm_requirement_chat_session.id`

关系图文字版：

- 一个需求 `rm_requirement`
  - 可以发起多次 AI 会话 `rm_requirement_chat_session`
- 一个会话 `rm_requirement_chat_session`
  - 理论上可以包含多条消息 `rm_requirement_chat_message`

当前代码现状：

- `rm_requirement_chat_session` 已实际使用。
- `rm_requirement_chat_message` 目前尚未写入。
- 这表示数据库结构已经为“完整会话持久化”预留了扩展位。

### 6.5 全局关系总览

如果把所有表连起来，可以理解成下面这张文字图：

1. `sys_user` 是大多数业务表的“操作者”和“负责人”来源
2. `pm_project` 隶属于用户
3. `rm_requirement` 隶属于项目
4. `rm_requirement_version` 隶属于需求
5. `rm_requirement_chat_session` 也隶属于需求
6. `rm_requirement_chat_message` 隶属于会话

也就是说，主业务主线是：

`用户 -> 项目 -> 需求 -> 版本`

以及一条并行的 AI 会话线：

`用户 -> 需求 -> AI会话 -> 会话消息`

## 7. 结合代码与表结构的理解建议

在后续阅读和开发时，可以用下面这套映射方式快速定位：

1. 先确认页面属于哪个业务域
2. 再定位对应 Controller
3. 再看 Service 是否只做编排，还是同时承担规则
4. 再看 Mapper 或 XML，确认最终落在哪张表

例如：

- 页面上“创建需求”
  - 看 `RequirementController`
  - 再看 `RequirementService.createRequirement(...)`
  - 再看 `RequirementMapper.insert(...)`
  - 最终落表 `rm_requirement`

- 页面上“生成 PRD”
  - 看 `RequirementDocgenController.generate(...)`
  - 再看 `RequirementDocgenService.generate(...)`
  - 再看 `DocGenService.generatePrdFromChat(...)`
  - 最后回到 `RequirementService.addVersion(...)`
  - 最终落表 `rm_requirement_version`

## 10. 当前权限矩阵总结

下面用角色视角做一个简化总结。

### 10.1 ADMIN

- 可访问所有受保护接口
- 原因是 `AuthInterceptor` 中对 `ADMIN` 角色有超管放行逻辑

### 10.2 普通登录用户

默认可访问：

- 获取当前用户信息
- 查看项目列表与项目详情
- 查看项目下需求列表
- 查看需求详情
- 通用 AI 工作台接口

但若没有显式权限，则不能访问：

- 创建/编辑项目
- 创建/编辑需求
- 查看需求版本
- 参与需求 AI 工作台
- 生成 PRD
- 系统管理

### 10.3 推荐的后续角色设计思路

如果后面要完善权限体系，可以考虑拆成如下角色：

1. 项目管理员
   - 具备 `PROJECT:CREATE`、`PROJECT:EDIT`
2. 需求分析师
   - 具备 `REQUIREMENT:CREATE`、`REQUIREMENT:EDIT`、`REQUIREMENT:AI_CHAT`
3. 审批/发布角色
   - 具备 `REQUIREMENT:GENERATE_PRD`、`REQUIREMENT:VERSION_VIEW`
4. 系统管理员
   - `ADMIN` 或受控的系统管理权限组合
