## 设计方案文档（通用模板）

**项目名称**：{{项目名称}}  
**文档版本**：{{版本号}}  
**日期**：{{日期}}  

### 1. 总体架构
1. 前端（Vue）
   - 页面：项目/任务入口、输入业务描述、任务状态列表、文档预览与下载。
   - 交互：通过统一API调用后端，使用 `traceId` 贯穿请求链路。

2. 后端（Spring Boot）
   - 模块：鉴权/权限、任务编排、提示词编排、结构化校验、文档渲染、导出服务、审计日志。
   - 输出：统一响应体 `{ code, message, data, traceId }`。

3. Agent/LLM层
   - 输入：用户业务描述 + 选择模板/版本 + 上下文（可选）。
   - 输出：分阶段结构化草稿（JSON）-> 校验 -> 最终Markdown片段 -> 汇总成文档集。

### 2. 模块设计
1. 模板中心（Template Center）
   - 模板版本：`template_version`（可配置默认模板）。
   - 占位符渲染：将 `{{项目名称}}/{{版本号}}/...` 等占位符替换为运行时数据。
   - 渲染策略：Markdown拼接规则、表格字段一致性约束。

2. 任务编排（Job Orchestration）
   - 状态机（示例）：
     - `PENDING`（待开始）
     - `CLARIFYING`（澄清中）
     - `GENERATING`（生成中）
     - `VALIDATING`（校验中）
     - `COMPLETED`（完成）
     - `FAILED`（失败）
   - 重试策略：仅对“可恢复错误”（如LLM格式错误）重试，业务错误不重试。

3. 文档生成服务（Doc Generation）
   - 生成对象：
     - `01-PRD-Agent需求文档.md`
     - `02-设计方案文档.md`
     - `03-技术方案文档.md`
     - `04-功能清单.md`
     - `05-接口说明.md`
     - `06-数据表设计.md`
     - `07-验收标准.md`
   - 生成流程（建议固定）：
     - 需求澄清 -> 草稿JSON -> 校验 -> Markdown渲染 -> 一致性检查（跨文档引用字段）

4. 文档校验与一致性（Validation & Consistency）
   - 输入校验：必填字段缺失 -> 阻塞并返回澄清问题。
   - 输出校验：
     - 表头字段必须匹配约定（如接口表格字段）
     - 每个接口必须有：ID/URL/Method/认证/请求/响应/错误码约定
     - 数据表必须与接口字段可落库（字段类型/必填/主键策略一致）

5. 导出与回溯（Export & Trace）
   - 导出：Markdown转Word/PDF（可选能力）
   - 回溯：展示生成输入快照、每阶段耗时、traceId、LLM原始片段（可选审计）

### 3. 数据流与时序（文字版）
1. 前端提交：`createDocGenJob(业务描述, 模板选择, 选项)`
2. 后端创建任务记录（写库），返回 `jobId`
3. 后端执行任务编排：
   - 阶段1：生成澄清问题（若缺失信息 -> 返回给前端）
   - 阶段2：拿到补齐信息后生成各文档
   - 阶段3：结构化校验失败 -> 重试或返回明确错误
4. 完成后：存储生成结果（Markdown/HTML/文件元数据）
5. 前端：查询任务列表、拉取生成结果、执行导出（可选）

### 4. 关键设计决策
1. 输出格式统一策略
   - 所有文档均使用固定标题/固定表头，便于校验与验收。

2. 错误处理策略
   - 对“格式/结构不合规”执行结构化重试。
   - 对“业务信息不足”阻塞并返回澄清问题列表（非硬失败）。

3. 一致性约束
   - 接口说明与数据表设计之间通过“字段来源映射”保证落库可行。
   - 验收标准必须覆盖功能清单用例（可测试口径）。

### 5. 可观测性设计
- `traceId`：贯穿前端请求、后端服务、LLM调用、文档渲染、导出链路。
- 日志：
  - 任务生命周期日志
  - 每阶段耗时
  - 校验失败原因（可结构化）
- 指标（可选）：
  - 生成成功率
  - p95生成耗时
  - 校验失败类型分布
