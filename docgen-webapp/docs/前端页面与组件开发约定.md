## 开工前必看

每次开始前端页面工作，先按这个顺序看：

1. 看路由和页面归属，确认是新页面还是已有页面扩展
2. 看最接近的领域目录，确认有没有可以复用的组件模式
3. 看页面应该由 `view` 承担什么、由 `components/<domain>` 承担什么
4. 写完后跑一次前端构建

如果只是改按钮文案或小样式，可以简化处理；如果涉及页面结构、导航、状态流转，就不要跳过这一步。

## 页面怎么写

页面层，也就是 `src/views/*.vue`，默认只做这些事：
- 处理路由参数和页面跳转
- 拉数据、提交流程、错误提示
- 管理跨多个卡片共享的状态
- 把页面拼成完整布局

页面层不要再做这些事：
- 把所有模板都堆在一个文件里
- 自己重复实现全局导航、面包屑、返回区
- 到处内联状态文案、枚举映射、重复格式化逻辑

一句话：

`view` 负责编排，`components/<domain>` 负责内容区块。

## 组件怎么拆

当一个页面出现下面任一情况，就应该考虑拆组件：
- 页面里已经有多个业务区块
- 同一页里既有表单、又有列表、又有详情、又有弹层
- 同样的状态文案、标签映射、数据展示逻辑反复出现
- 页面文件已经开始难读、难改、难定位

当前项目推荐的拆分方式是：
- 可见区块：`XxxCard.vue` / `XxxPanel.vue`
- 弹层：`XxxModal.vue`
- 共享类型：`types.ts`
- 格式化、枚举文案、选项映射：`helpers.ts` 或 `labels.ts`

不要一上来就造一个特别抽象的“万能公共组件”。
先按领域拆，领域内复用稳定后，再考虑往更公共的层级抽。

## 状态怎么放

当前项目没有上 Pinia / Vuex，默认做法是：
- 页面局部状态用 `ref`
- 成组表单用 `reactive`
- 派生结果用 `computed`
- 同步副作用用 `watch`

状态放置规则：
- 只在一个区块用的状态，放当前组件
- 多个卡片共用的状态，放父级 `view`
- 路由身份相关状态，从 `useRoute()` 推导
- 登录态和 token，统一在 `src/auth.ts`
- 少量需要刷新后保留的草稿，可以放 `localStorage`

不要把一个页面的选中态同时放在父组件和多个子组件里。
谁是源头，要非常明确。

## 什么时候抽 composable

当前项目还没有大规模的 `useXxx` 组合式目录，所以不要为了“看起来高级”提前抽一堆 composable。

目前默认规则是：
- 单页内的业务流程，先放在页面层
- 纯格式化逻辑，放 `helpers.ts`
- 多个文件共享的数据结构，放 `types.ts`
- 真的出现跨页面、跨领域复用的状态逻辑时，再考虑抽 `useXxx.ts`

也就是说，先让结构清晰，再做抽象。

## 类型怎么放

当前推荐做法：
- auth 相关类型，放在 `src/auth.ts`
- 某个领域共享的类型，放在该领域目录下的 `types.ts`
- 只在一个组件用的小类型，可以先放本文件

不要把同一个对象结构在多个页面里复制粘贴。
一旦同领域两个文件以上共用，就提到 `types.ts`。

同时要注意：
- 路由参数要显式转成数字或需要的类型
- `localStorage` 读取要做 `try/catch`
- 后端枚举不要直接裸显示给用户，要经过文案映射

## 文案和标签怎么处理

用户能看到的内容，优先使用中文一致口径。

比如这些不要直接出现在模板里：
- `PRIVATE`
- `ACTIVE`
- `ARCHIVED`
- `P1`
- `OWNER`

应该通过领域内 `helpers.ts` / `labels.ts` 统一映射后再显示。

这样做的好处是：
- 页面更干净
- 文案统一
- 后面改口径时只改一处

## 导航怎么处理

全局导航、面包屑、上下文栏属于壳层能力，应该由：
- `src/components/AppShell.vue`
- `src/components/shell/*`

来承接。

页面本身不要再重复做一层顶部导航，否则很容易出现：
- 双标题
- 双返回
- 双导航条

如果页面需要的是“页内功能入口”或“快捷操作”，应放在内容区卡片里，而不是重新造一层全局导航。

## 构建验证

只要涉及这些改动之一，就必须重新跑前端构建：
- 页面结构调整
- 组件拆分
- 路由修改
- 公共 helper / types 变更
- 导航和壳层调整

当前标准命令：

```powershell
wsl.exe bash -lc '. "$HOME/.nvm/nvm.sh"; nvm use default >/dev/null; cd /mnt/g/Agent/docgen-webapp/frontend && npm run build'
```

不要因为编辑器里没报红，就跳过构建。

## 常见反模式

下面这些做法，后续默认不要再出现：

- 把一个完整页面重新写成几百行单文件
- 在多个模板里重复写同一套状态枚举文案
- 页面自己实现一层全局导航
- 明明只是领域内复用，却直接造跨项目“万能组件”
- 改完结构不跑构建

## 推荐开发流程

后续写页面，建议固定按这个顺序：

1. 看本文件和 `.trellis/spec/frontend/*`
2. 找一个最接近的现有页面做参考
3. 先定 `view` 和 `components/<domain>` 的分工
4. 再写模板、状态、交互
5. 提炼 `types.ts` / `helpers.ts`
6. 跑构建
7. 如果形成了新模式，顺手更新规范

## 最后一句

以后前端开发的默认原则是：

先看规范，再起页面；
先按领域拆，再谈抽象复用；
先保证可构建，再继续扩展。
