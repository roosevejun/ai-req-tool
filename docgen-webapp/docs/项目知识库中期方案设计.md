# 项目知识库中期方案设计

## 1. 目标

本文档定义“项目知识库”中期方案，目标是把当前“AI 创建项目”里的资料能力，从“会话内临时参考资料”升级为“可沉淀、可检索、可复用的项目知识库”。

方案重点解决以下问题：

- URL 资料目前只保存链接，不能真正抓取和学习网页内容
- 文本资料虽然能保存，但缺少统一的知识文档抽象
- 后续会有越来越多的非结构化数据，不能长期只放数据库正文列
- AI 对话、项目创建、需求整理后续都需要统一的知识检索入口

中期方案原则：

- 结构化数据继续保存在 PostgreSQL
- 原始非结构化内容迁移到对象存储
- AI 可消费内容沉淀为“知识文档 + 文档切片”
- 会话、项目、需求都通过知识库进行复用

---

## 2. 当前现状

当前已经具备的基础能力：

- `pm_project` 已有较完整的项目结构化字段
- `pm_project_chat_session` / `pm_project_chat_message` 已支持项目 AI 会话
- `pm_project_source_material` 已支持保存 URL / TEXT 资料
- `ProjectAiConversationService` 已能把资料摘要拼进提示词

当前主要限制：

- URL 资料只保存 `source_uri`，没有抓取正文
- 文本资料虽然保存 `raw_content`，但没有统一知识文档生命周期
- 没有异步处理状态、解析任务、失败重试
- 没有切片、embedding、召回检索能力

---

## 3. 总体架构

中期方案建议拆成四层：

1. 对象存储层
2. 知识文档元数据层
3. 内容处理流水线
4. 检索增强层

### 3.1 对象存储层

用于保存原始非结构化内容与处理结果文件。

建议存储内容：

- 网页抓取得到的原始 HTML
- 清洗后的纯文本
- 用户上传的 PDF / Word / 图片 / 音频
- OCR 结果
- 转写结果

推荐技术：

- MinIO
- 或 S3 兼容对象存储

数据库中只保存引用信息：

- `storage_bucket`
- `storage_key`
- `mime_type`
- `size_bytes`
- `sha256`

### 3.2 知识文档元数据层

PostgreSQL 负责保存强结构化元数据、索引与业务关联。

核心思想：

- 一个“资料”升级为一个“知识文档”
- 文档可以属于项目，也可以进一步关联到需求
- 文档可以有多个处理产物
- 文档切片独立管理，便于后续检索

### 3.3 内容处理流水线

所有 URL / TEXT / FILE 资料统一进入异步处理流程。

处理链路：

1. 创建知识文档记录
2. 进入待处理状态
3. 根据类型执行解析
4. 文本清洗与标准化
5. AI 摘要与标签抽取
6. 文本切片
7. embedding 写入向量索引
8. 文档状态切换为可检索

### 3.4 检索增强层

供 AI 创建项目、AI 需求整理、需求工作台统一调用。

最小能力：

- 按项目范围过滤文档
- 按需求范围过滤文档
- 关键词/标签辅助过滤
- 向量召回相关 chunk
- 将命中的 chunk 组装为 LLM 上下文

推荐技术：

- PostgreSQL + pgvector
- 或独立向量库 Qdrant / Milvus

中期优先建议：

- 如果部署复杂度优先，先用 `pgvector`
- 如果规模和召回能力优先，再切换独立向量库

## 4. 数据模型设计

## 4.1 新增表建议

建议新增以下核心表：

- `km_document`
- `km_document_asset`
- `km_document_chunk`
- `km_document_task`

可选后续表：

- `km_document_tag`
- `km_retrieval_log`

### 4.1.1 km_document

用途：

- 知识文档主表
- 管理资料来源、归属关系、处理状态和摘要信息

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGSERIAL PK | 主键 |
| `project_id` | BIGINT NULL | 归属项目 |
| `requirement_id` | BIGINT NULL | 归属需求，可空 |
| `source_material_id` | BIGINT NULL | 对历史 `pm_project_source_material` 的映射，可空 |
| `document_type` | VARCHAR(16) | `URL` / `TEXT` / `FILE` |
| `source_uri` | VARCHAR(1000) NULL | 原始链接 |
| `title` | VARCHAR(256) NULL | 标题 |
| `status` | VARCHAR(32) | `PENDING` / `PROCESSING` / `READY` / `FAILED` / `DISABLED` |
| `raw_text` | TEXT NULL | 小文本场景下可直接存数据库 |
| `clean_text` | TEXT NULL | 清洗后文本，小体量时可内存数据库 |
| `summary` | TEXT NULL | AI 摘要 |
| `keywords_json` | TEXT NULL | 关键词 JSON |
| `tags` | VARCHAR(512) NULL | 轻量标签 |
| `content_hash` | VARCHAR(64) NULL | 内容 hash，用于去重 |
| `version_no` | INT | 文档版本 |
| `is_latest` | BOOLEAN | 是否最新版本 |
| `retrievable` | BOOLEAN | 是否可参与检索 |
| `created_by` | BIGINT | 创建人 |
| `created_at` | TIMESTAMP | 创建时间 |
| `updated_at` | TIMESTAMP | 更新时间 |

说明：

- 小文本可以先直接保留在 `raw_text/clean_text`
- 大内容建议写对象存储，并在 `km_document_asset` 中记录

### 4.1.2 km_document_asset

用途：

- 文档关联的文件或对象存储产物

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGSERIAL PK | 主键 |
| `document_id` | BIGINT | 所属文档 |
| `asset_role` | VARCHAR(32) | `ORIGINAL` / `CLEAN_TEXT` / `OCR_TEXT` / `HTML` / `PDF` |
| `storage_bucket` | VARCHAR(128) | 存储桶 |
| `storage_key` | VARCHAR(512) | 对象 key |
| `mime_type` | VARCHAR(128) NULL | MIME 类型 |
| `size_bytes` | BIGINT NULL | 文件大小 |
| `sha256` | VARCHAR(64) NULL | 文件 hash |
| `created_at` | TIMESTAMP | 创建时间 |

### 4.1.3 km_document_chunk

用途：

- 文档切片
- 检索的最小单位

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGSERIAL PK | 主键 |
| `document_id` | BIGINT | 所属文档 |
| `chunk_no` | INT | 切片序号 |
| `chunk_text` | TEXT | 切片正文 |
| `token_count` | INT NULL | token 数 |
| `summary` | TEXT NULL | 切片摘要，可选 |
| `embedding_status` | VARCHAR(32) | `PENDING` / `READY` / `FAILED` |
| `vector_ref` | VARCHAR(128) NULL | 向量库主键或引用 |
| `created_at` | TIMESTAMP | 创建时间 |

### 4.1.4 km_document_task

用途：

- 跟踪异步处理状态

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | BIGSERIAL PK | 主键 |
| `document_id` | BIGINT | 所属文档 |
| `task_type` | VARCHAR(32) | `FETCH_URL` / `PARSE_FILE` / `EXTRACT_TEXT` / `SUMMARIZE` / `CHUNK` / `EMBED` |
| `status` | VARCHAR(32) | `PENDING` / `RUNNING` / `SUCCESS` / `FAILED` |
| `attempt_count` | INT | 重试次数 |
| `last_error` | TEXT NULL | 最近错误 |
| `started_at` | TIMESTAMP NULL | 开始时间 |
| `finished_at` | TIMESTAMP NULL | 结束时间 |
| `created_at` | TIMESTAMP | 创建时间 |

## 4.2 与现有表的关系

### 4.2.1 与 pm_project_source_material 的关系

中期不建议直接删除现有表，建议分阶段处理：

- 第一步：保留 `pm_project_source_material`
- 第二步：新增 `km_document`
- 第三步：新创建资料优先写 `km_document`
- 第四步：老数据迁移后，`pm_project_source_material` 退化为兼容层或废弃

迁移期可以保留：

- `source_material_id -> km_document.id` 的映射
- 或 `km_document.source_material_id`

### 4.2.2 与项目和需求的关系

建议知识文档支持两级归属：

- 项目级知识：`project_id` 有值，`requirement_id` 为空
- 需求级知识：`project_id` 和 `requirement_id` 都有值

这样能支持：

- AI 创建项目阶段沉淀项目级知识
- 后续需求澄清阶段沉淀需求级知识
- 项目级知识复用给多个需求

## 5. 处理流程设计

## 5.1 URL 资料

目标：用户添加一个网址后，系统能真正“学习网页内容”。

流程：

1. 前端提交 URL
2. 后端创建 `km_document`
3. 创建 `FETCH_URL` 任务
4. 异步抓取网页 HTML
5. 提取正文
6. 清洗导航、广告、脚本
7. 生成摘要 / 关键词 / 标签
8. 切片
9. embedding
10. 状态切换为 `READY`

注意点：

- 要限制域名白名单/黑名单
- 要限制抓取超时和页面大小
- 要处理抓取失败、403、重定向、反爬

## 5.2 TEXT 资料

流程：

1. 保存原始文本
2. 标准化换行与空白
3. AI 摘要
4. 切片
5. embedding
6. 状态切换为 `READY`

## 5.3 FILE 资料

流程：

1. 文件上传到对象存储
2. 创建 `km_document`
3. 根据 MIME 类型选择解析器
4. 提取文本
5. 必要时 OCR
6. 摘要 / 切片 / embedding
7. 状态切换为 `READY`

---

## 6. 检索策略设计

## 6.1 检索范围

默认按当前场景限制范围：

- AI 创建项目：只检索当前会话/当前项目的资料
- 项目需求整理：优先检索项目级知识 + 当前需求级知识
- 需求工作台：优先当前需求，补充项目级知识

## 6.2 检索过程

建议检索顺序：

1. 范围过滤
2. 状态过滤：`READY` + `retrievable = true`
3. 关键词或标签过滤
4. 向量召回 topK chunk
5. 重排
6. 拼接上下文给 LLM

## 6.3 召回返回格式

建议统一返回：

- 文档标题
- 来源 URL / 文件名
- chunk 文本
- chunk 摘要
- 相关分数
- 所属项目 / 需求

- `KnowledgeDocumentService`
- `KnowledgeIngestService`
- `KnowledgeRetrievalService`
- `KnowledgeTaskService`

### 7.1 KnowledgeDocumentService

职责：

- 创建知识文档
- 查询文档
- 更新文档状态
- 管理版本

### 7.2 KnowledgeIngestService

职责：

- URL 抓取
- 文件解析
- 文本清洗
- 摘要抽取

### 7.3 KnowledgeRetrievalService

职责：

- 按项目/需求召回知识
- 输出给 AI 会话的上下文

### 7.4 KnowledgeTaskService

职责：

- 管理异步任务
- 重试
- 失败告警

## 8. 与现有 AI 创建项目流程的对接

当前 `ProjectAiConversationService` 可以做如下演进：

### 当前方式

- 直接读取 `pm_project_source_material`
- 把摘要拼到 `description` 后面

### 中期方式

- 会话添加资料时，先落 `km_document`

- `guideProjectProductInfo(...)` 不再只吃“资料摘要拼接字符串”

- `GET /api/requirements/{requirementId}/knowledge`

## 9. 迁移路径

建议按三阶段推进。

### 阶段一：建模与兼容

目标：

- 新增 `km_document` 等表
- 不改旧功能路径
- 新代码支持双写

要做的事：

- 建表脚本
- Mapper / Entity / Service
- 新资料入库时同步写 `pm_project_source_material` 和 `km_document`

### 阶段二：资料处理异步化

目标：

- URL 真正抓取正文
- 文本 / 文件统一处理

要做的事：

- 异步任务执行器
- URL 抓取器
- 文件解析器
- 文档状态机

### 阶段三：RAG 接入

目标：

- 项目和需求对话都走知识检索

要做的事：

- 切片与 embedding
- 检索服务
- AI 会话上下文改造

## 10. 风险与注意事项

### 10.1 数据膨胀

风险：

- 全文、切片、embedding 都放数据库会迅速膨胀

建议：

- 原始大文件放对象存储
- 数据库只保索引和必要文本

### 10.2 权限隔离

风险：

- 不同项目/需求资料混检索，可能造成越权

建议：

- 检索层必须按项目、需求、用户权限做过滤

### 10.3 资料质量不稳定

风险：

- URL 正文抓取质量差
- OCR 误差高

建议：

- 保存原文
- 保存清洗结果
- 允许人工修正摘要和标签

### 10.4 成本控制

风险：

- 大规模 URL 抓取、embedding 和摘要有成本

建议：

- 只对 `READY_FOR_AI` 资料做完整处理
- 支持延迟 embedding

---

## 11. 建议优先级

如果按收益/成本比排序，建议这样做：

1. 先建 `km_document` / `km_document_chunk` / `km_document_task`
2. 接 URL 抓取和文本清洗
3. 打通 AI 创建项目读取知识文档摘要
4. 再做 chunk + embedding
5. 再做需求工作台 / 需求整理统一检索
